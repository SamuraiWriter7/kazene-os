## Royalty OS Core Loop

Royalty OS is structured around three layers:

### Permission
Defines what may be used and under what conditions.

Examples:
- machine-readable usage conditions,
- scope of use,
- policy versioning,
- licensing terms.

### Trace
Records what was used, when, and how.

Examples:
- source access,
- retrieval logs,
- response-linked evidence,
- contribution-related trace data.

### Allocation
Determines how recorded value should be distributed or returned.

Examples:
- royalty distribution,
- usage-based return,
- auditable allocation events,
- return calculations grounded in trace.

Trace Architecture

Trace is the key interface between KazeneOS and Royalty OS.

It should not be treated as a single undifferentiated log.
At minimum, it can be divided into three layers:

Access Trace

Records what was accessed.

Examples:

which document,
which chunk,
which source,
when it was retrieved,
for which query or response.
Influence Trace

Estimates what affected the output or behavior.

Examples:

reflected summaries,
semantic contribution,
approximate influence from retrieved material,
contribution estimates used for return.
Audit Trace

Records who guarantees the record and under what conditions.

Examples:

signatures,
verification events,
revision history,
audit metadata,
dispute-related review history.

In short:

Access Trace = what was accessed
Influence Trace = what affected the output
Audit Trace = who guarantees the record

This layered trace structure allows return systems to be more realistic, more auditable, and less dependent on the illusion of perfect attribution.

Why this structure matters

Without an upper-layer structure like KazeneOS:

return systems become isolated,
trace becomes a narrow logging artifact,
governance becomes reactive,
value circulation loses philosophical coherence.

Without a practical subsystem like Royalty OS:

KazeneOS risks remaining too abstract,
return remains undefined,
trace remains conceptually rich but operationally weak.

Together, they solve opposite problems:

KazeneOS prevents reduction
Royalty OS prevents vagueness

That is why the relationship is not one of competition, but of hierarchy and specialization.

Subsystem View

KazeneOS can host multiple circulation-oriented subsystems.

Royalty OS

Formal value return through Permission, Trace, and Allocation.

Gratitude OS

Non-monetary return through acknowledgment, gratitude, and symbolic circulation.

Trust OS

Reliability and confidence accumulation across agents, records, and interactions.

This shows that value circulation in the AI era is not reducible to money alone.
It may also take symbolic, relational, and reputational forms.

Practical Reading

This repository should be read as an upper-layer architecture repository.

It does not attempt to be:

a universal payment network,
a complete legal framework,
a finished attribution engine,
a production-ready protocol stack.

Instead, it defines the structural map within which such systems may later be designed.
